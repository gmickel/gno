# Autonomous Retrieval Research Harness

Constrained experiment loop for the `research/finetune/` sandbox.

Scope:

- mutate sandbox files only
- fixed budget
- fixed validation/promotion metric
- human promotion gate before any runtime default changes

## Files

```text
research/finetune/autonomous/
├── config.json
├── policy.md
├── runs/
└── scripts/
    └── noop-run.ts
```

## Loop

1. read `policy.md` and `config.json`
2. mutate a tiny set of sandbox files
3. run one short experiment against the fixed metric
4. log result + changed files + decision
5. keep or discard locally

## Hard Boundaries

Allowed:

- `research/finetune/configs/**`
- `research/finetune/contracts/**`
- `research/finetune/data/training/**`
- `research/finetune/scripts/**`
- `research/finetune/autonomous/**`

Disallowed:

- `src/**`
- `docs/**`
- `spec/output-schemas/**`
- `evals/fixtures/**` except benchmark artifacts already generated by humans
- any product/runtime defaults

## Fixed Metric

- baseline reference: `evals/fixtures/retrieval-candidate-benchmark/latest.json`
- preflight command: `bun run research:finetune:validate`
- smoke command: `bun test test/research/finetune-sandbox.test.ts`
- promotion split: `heldout`

## Noop Proof

Run:

```bash
bun run research:finetune:autonomous:noop
```

The noop run must:

- write exactly one run artifact under `research/finetune/autonomous/runs/`
- leave the rest of the repo unchanged by the harness itself
- fail if any new mutation escapes the allowed roots

## Evaluation Cycle

For an actual sandbox evaluation cycle:

```bash
bun run research:finetune:autonomous:evaluate mlx-run1
```

This will:

1. validate the declared mutation targets are inside the sandbox
2. call the promoted run pipeline for the named run
3. compare benchmark deltas vs the shipped baseline
4. emit a keep/discard decision under `autonomous/runs/`

The human gate remains explicit. The script does not change product defaults.
