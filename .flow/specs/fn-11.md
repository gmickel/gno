# fn-11 Setup React component testing infrastructure

**Migrated from:** gno-lch
**Original type:** task
**Priority:** P2

---

## Goal

Add a real interaction-level test harness for the web UI so GNO is no longer limited to:

- pure logic tests
- server-rendered markup snapshots via `renderToStaticMarkup`
- manual browser checks

The immediate goal is not broad test quantity. The goal is a stable first testing stack for interactive UI behavior.

## Current State

Today the repo already has many `bun:test` web tests, but they are mostly one of:

- pure helper logic tests
- static render checks using `react-dom/server`
- tests that explicitly avoid DOM interaction because there is no browser-like environment yet

Examples already in repo:

- `test/serve/public/components/FirstRunWizard.test.tsx`
- `test/serve/public/navigation.test.tsx`
- `test/serve/public/components/WikiLinkAutocomplete.test.tsx`
- `test/serve/public/connectors-page.test.tsx`

This means interactive behaviors like keyboard handling, focus management, menu/dialog behavior, form interactions, and accessibility assertions are still under-tested.

## Constraints

- Stay Bun-first.
- Prefer `bun:test`, not Jest/Vitest migration.
- Do not rewrite the whole suite.
- Preserve the existing lightweight static-render tests where they are still useful.
- Add DOM/browser-style tests only where interaction coverage is genuinely valuable.
- Keep the first E2E pass small and high-signal.

## Scope

### 1. DOM test harness

Add a DOM environment for interactive component tests.

Recommended stack:

- `bun:test`
- `@testing-library/react`
- `@testing-library/user-event`
- `happy-dom` or `jsdom`

Pick one environment and document why it fits Bun best.

Also add:

- shared test setup file(s)
- render helpers for common providers / globals
- a pattern for module mocking that works with the existing Bun test style

### 2. First interactive component coverage

Add DOM interaction tests for at least these high-value components:

- `src/serve/public/components/TagInput.tsx`
- `src/serve/public/components/TagFacets.tsx`
- one dialog- or command-style component with keyboard/focus behavior
  - likely `CaptureModal`, `QuickSwitcher`, or another similarly interactive surface

These should test real user behavior, not just string presence.

### 3. Page-level interaction coverage

Add higher-level DOM tests for the most interaction-heavy pages/surfaces.

Minimum target set:

- `src/serve/public/pages/Search.tsx`
- `src/serve/public/pages/DocView.tsx` or `src/serve/public/pages/DocumentEditor.tsx`
- one capture/create flow surface (`CaptureModal` or equivalent UI entry path)

Focus on:

- form state changes
- advanced retrieval control toggles
- navigation callbacks
- empty/loading/error states where relevant

### 4. Accessibility baseline

Add first-pass accessibility assertions for the new DOM-tested surfaces.

This does not need to become a full accessibility program in one task.
It does need to catch obvious regressions in:

- labels
- roles
- keyboard reachability
- dialog/listbox semantics

### 5. E2E/browser smoke path

Add a minimal browser E2E path.

Because Playwright is already present in devDependencies, the first pass should be:

- minimal repo-local setup
- one smoke flow only

Recommended smoke flow:

1. open the web UI
2. perform a search or browse interaction
3. open a document/capture surface
4. assert the main UI path is functional

Do not try to build a giant E2E matrix in this task.

## Non-goals

- replacing all existing static-render tests
- 100% UI coverage
- pixel/screenshot regression testing
- a large flaky end-to-end suite
- migrating the whole repo to a different test runner

## Deliverables

- DOM testing dependencies added and wired into `bun test`
- shared test setup/helpers for interactive React tests
- interaction tests for the targeted components/pages
- accessibility assertions for the new DOM-tested surfaces
- one browser E2E smoke path
- docs / task notes describing how to run the new test layers

## Acceptance

- [ ] `bun test` can run interactive React component tests in a real DOM-like environment.
- [ ] The chosen DOM setup is documented and reusable for future tests.
- [ ] `TagInput` and `TagFacets` have real interaction coverage.
- [ ] At least two high-value page/surface tests cover interaction, not just static markup.
- [ ] Accessibility assertions exist for the first DOM-tested surfaces.
- [ ] A minimal Playwright/browser smoke flow exists and is runnable.
- [ ] The task leaves the repo with a clear pattern another agent can extend without rethinking the stack.
