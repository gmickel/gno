# fn-109 CJK Lexical Normalization

## Goal & Context
<!-- scope: business -->

Improve Chinese, Japanese, and Korean retrieval when semantic search is unavailable, but only with a representation proven by `fn-96`. Ship the smallest deterministic normalization/tokenization change that materially improves each failing benchmark lane without harming Latin/identifier retrieval or unacceptable index cost.

## Architecture & Data Models
<!-- scope: technical -->

Use `fn-96` artifacts to select among benchmarked representations such as script-aware segmentation, character n-grams, Unicode normalization variants, or an FTS side representation. Keep original canonical text and source spans unchanged. The selected lexical representation is generated by one shared analyzer at index and query time and is fingerprinted by version/config.

Prefer an additive FTS column/table or tokenizer path that permits migration/backfill and rollback. Map matches back to original chunks/line ranges; normalized tokens never appear as user-visible snippets. Language detection chooses the analyzer conservatively, with mixed-script support and neutral fallback.

## API Contracts
<!-- scope: technical -->

- No new user-facing search command is required; existing BM25/hybrid surfaces improve transparently.
- Status/diagnose/explain add analyzer/fingerprint metadata and fallback warnings.
- Config may expose a controlled `lexicalAnalyzer` override only if benchmark evidence shows automatic detection is insufficient.
- Index schema/migration records analyzer version and triggers bounded reindex/backfill.

## Edge Cases & Constraints
<!-- scope: technical -->

- Preserve exact ASCII identifiers, URLs, file paths, numbers, emoji, punctuation, and mixed Latin/CJK terms.
- Handle simplified/traditional Chinese, kana/kanji, hangul, compatibility forms, combining marks, and malformed Unicode.
- Query/index analysis must be identical and deterministic.
- Avoid unbounded n-gram index growth and pathological query expansion.
- Existing indexes remain readable during migration or fail with explicit rebuild guidance.
- English/German/French/Italian and code fixtures cannot regress materially.

## Acceptance Criteria
<!-- scope: both -->

- **R1:** The implementation choice is justified by committed `fn-96` comparisons rather than competitor precedent or intuition.
- **R2:** Every CJK lane independently clears the per-language floors frozen in `evals/fixtures/cjk-lexical-benchmark/promotion-gates.json`: Chinese Recall@5/10, MRR, and nDCG@10 at least `0.3611`/`0.4722`/`0.377`/`0.3981` with zero-result at most `0.5278`; Japanese at least `0.375` for all four metrics with zero-result at most `0.625`; Korean at least `0.75` with zero-result at most `0.25`.
- **R3:** Existing Latin-language and code Recall@10/nDCG@10 lose at most `0.02` absolute; zero previously passing identifier cases or new identifier zero-results are allowed; phrase and mixed-script regression fixtures remain green.
- **R4:** Against a same-run production baseline, index size stays at most `1.75x`, build time at most `2x`, and warm-query p95 at most `3x` with no more than `2 ms` absolute increase.
- **R5:** Original snippets/line ranges remain exact; explain/diagnose exposes analyzer and fingerprint without normalized-text leakage.
- **R6:** Migration/backfill/rollback, stale fingerprint, offline mode, and cross-platform tests are complete.
- **R7:** Public multilingual documentation is updated with measured semantic and lexical results and limitations.

## Boundaries
<!-- scope: business -->

No embedding model change, translation/transliteration service, broad NLP dependency without benchmark proof, support claim for all Asian languages, or user-visible text rewriting.

## Decision Context
<!-- scope: both — conditionally substructured -->

### Motivation
<!-- scope: business -->

Multilingual reliability includes degraded local lexical operation, not only the happy path with downloaded embeddings.

### Implementation Tradeoffs
<!-- scope: technical -->

Benchmark-selected additive representation costs index space but preserves source fidelity and rollback. A production change is justified only if per-language gains exceed its size/latency burden.

## Early proof point

Task `fn-109-cjk-lexical-normalization.1` validates the core approach (fn-96 artifacts select one representation that clears per-language lift and cost gates before production schema work).
If it fails, re-evaluate whether any production lexical change is justified before continuing with `fn-109-cjk-lexical-normalization.2`+.

## Outcome — no ship

The early proof point failed. The strongest committed diagnostic,
`substring-nfc`, misses the frozen Chinese Recall@10 floor (`0.4444 <
0.4722`), exceeds the zero-result ceiling (`0.5556 > 0.5278`), and recovers
only two of the three required additional hits. Neither diagnostic has the
production non-regression, index/build, cross-platform, or rollback evidence
required for selection.

The immutable decision receipt is
`evals/fixtures/cjk-lexical-benchmark/candidates/2026-07-22-no-ship.json`.
No representation was selected, no threshold was weakened, and no production
source, schema, packaging, or public support claim changed. Tasks 2–4 close as
intentionally not executed under this spec's early-proof contract. A future
attempt requires a new candidate comparison against the same frozen gates.
